# 진행도 — 테스트셋 300 확장 · 라벨 감사 (2026-07-21)

> 소스: `crawler/data/testset_natural_300.jsonl`(확장) → `testset_natural_300_v2.jsonl`(라벨 교정)
> 평가: `crawler/eval.py`(+오류분석 리포트 신규) · 코퍼스 dataset2(36문서 / 211청크)
> 한 줄 요약: 평가셋을 150 → 300으로 늘리고, 오류 원인 분석 도구를 붙여 라벨 23건을 근거 기반으로 교정 → hybrid 미적중 83 → 77건.

## I. 테스트셋 확장 (150 → 300)

### 배경
기존 `testset_natural.jsonl`은 150건이나 업무 편중이 심했다(착오송금 44 vs 채무조정 10, 4.4배). 또 대부분 FAQ·heading 문항이라 실제 사용자의 짧은/구어체 질의를 반영하지 못했다.

### 방침 (5개 조건)
1. 업무 영역 정규화 — 특정 파트 쏠림 완화
2. 기존과 의미는 같되 다양한 표현
3. 정답(gt_docs) 정확 매핑
4. 더 짧은 문장의 질문 추가
5. 저수준·단순 질의 추가 (예: "돈 잘못 보냈어요 어떡하죠?")

### 방법
- GitHub 코퍼스(`chunks.jsonl`)와 기존 150건의 **검증된 질문→gt_docs 매핑**을 근거 은행으로 사용.
- 신규 150건 생성: 각 문항에 `origin_question`(원 의미) 명기, gt_docs는 코퍼스 doc_id로만 매핑.
- 유형: short 89 · simple 44 · paraphrase 17.

### 결과
- 총 **300건** (기존 150 유지 + 신규 150).
- 업무 분포 평탄화: 착오송금 72 · 예금자보호 54 · 채무조정 46 · 미수령금 44 · 은닉재산 44 · 예금보험금 40 (최다:최소 4.4배 → 1.8배).
- 신규 평균 글자수 17.7자(기존 32.0자) — 짧은 문장 조건 충족.
- 중복 검증: 완전·표기 중복 0건. 유사도 0.75+ 쌍은 의도된 짧은·구어체 변형(격식체/구어체 강건성 테스트용)이며, 실질 중복 1건은 교체 처리.

## II. 오류 분석 도구 추가

### 문제
`eval.py`는 종합 지표(hit@1/hit@3/MRR)만 출력해, **어떤 문항이 왜 틀렸는지**를 알 수 없었다.

### 해결
- `eval.py`에 `write_error_analysis()` 추가 → `data/error_analysis.md` 자동 생성 (기존 `eval_report.md`는 유지).
- 미적중(top-3 밖) 문항마다 top-3를 태깅: **✅정답 / ⚠️타업무혼입 / ❌동일업무오답**.
- 부가 표: 업무별 미적중률 · 혼입 문서 랭킹 · 업무 간 혼입 흐름.
- 보조 스크립트 `compare_modes.py` — dense vs hybrid 문항별 승패 비교.

### 결과 (교정 전, 300건 기준)
- hybrid 미적중 **83건 (27.7%)**.
- 원인이 반반으로 갈림: 타업무혼입(top-1 다른 업무) 36건 · 동일업무오답 47건.
- **블랙홀 발견**: `selectFaqNramtAply`(미수령금 FAQ, 38청크)가 미적중 43문항의 top-3에 침투(총 61회). 보호한도·예금보험금 등 타 업무 질의까지 잠식.

## III. 라벨 감사·교정 (근거 기반)

### 방침
gt_docs = 질문 의도를 **본문에 실제로 답하는 문서 전부**. 리트리버 top-3에 들었다는 이유가 아니라 코퍼스 본문 내용으로 판단. "라벨은 정확한데 리트리버가 못 찾은 것"은 손대지 않음.

### 주요 교정 (총 23건: 채무조정 17 · 착오송금 4 · 은닉재산 1 · 예금보험금 1)
- **착오송금 금액/기한** 4건 → `selectFaqMsdrGvbkAply`(첫 문답 "5만원~1억, 1년 이내") 정답 추가.
- **채무조정 제도/서류/거절** 11건 → `LbltInfoInqItrdMthdGudn`(본문이 채무조정제도+구비서류) 추가.
- **채무정보조회** 6건 → 오라벨 교정. `LbltInfoInqItrdMthdGudn`은 제목만 "채무정보조회"일 뿐 **실제 파싱 본문은 채무조정제도**라, 조회 질의에선 제거하고 실제 조회 문서(`LbltInfoInqDscsnAply` + `selectFaqLbltInfoInq`)로 교체.
- **은닉 취득경위** 1건 → 취득경위 문답이 있는 `SprtFndCncmDclrFaq`로 교정.
- **예금보험금 창구** 1건 → 지급대행지점 방문 언급이 있는 `DpsmIbamtExpln` 추가.

산출: `testset_natural_300_v2.jsonl` · 변경 로그 `label_audit.md`.

### 결과 (교정 후)
- hybrid 미적중 **83 → 77건 (27.7% → 25.7%)**, hit@3 약 0.723 → 0.743.
- 업무별 미적중: 미수령금 20 · 예금보험금 15 · 착오송금 21 · 예금자보호 9 · 채무조정 7 · 은닉재산 5.

### 핵심 결론
- 교정된 23건 중 **실제 미적중은 10건, 라벨 교정만으로 회복된 건 ~6~7건**.
- 즉 **83개 오답 중 라벨 탓은 ~7개, 나머지 ~76개는 진짜 검색 문제**. "혼입부터 해결" 판단이 데이터로 확증됨.
- 부수 발견: `LbltInfoInqItrdMthdGudn`의 제목↔본문 불일치는 **파서 단계에서 잘못된 섹션을 추출했을 가능성** → 파이프라인 점검 필요.

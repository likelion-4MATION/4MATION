## 1. 청크 재태깅 + business_function 하드필터 (0.747 → 0.790)

**문제**: 미적중 83건 중 라벨 탓은 ~7건, 나머지는 검색 문제(「정리 1」§4). 특히 `selectFaqNramtAply`(미수령금 FAQ)가 미적중 43문항 top-3에 61회 침투.

**원인**: 이 페이지는 여러 업무 Q&A를 담은 cross-topic 게시판인데 **URL 규칙상 업무 1개(미수령금)로 태깅** → 청크 100% 오태깅. 게다가 38청크 = 인덱스에서 "복권 38장"(문서 크기 불균형). 청킹 자체는 정상.

**해결**:
- `chunk.py` 청크별 재태깅(결정론 키워드, 이중 게이트: 타 업무가 margin 2+ 우세할 때만 재태깅 / 문서 재태깅 비율 50% 미만이면 통째로 원 태그 유지). `selectFaqNramtAply` 미수38→예금보험28·예보6·미수4, `selectFaqTop10` 공통10→착오9·공통1. 텍스트 불변이라 재임베딩 없이 메타 태그만 갱신.
- `rag.py` `classify_query_bf()`(질의 키워드 분류, 커버리지 70%·정확도 99%, 불명이면 None→전체검색 폴백) + `hybrid_bf` 모드(예측 업무 청크로 검색 제한).

**선택 이유**: LLM 미개입 결정론 규칙(관리자 트리거 재생성 요건). 하드필터를 옵트인 모드로 추가해 기존 hybrid 보존.

**결과**: hybrid_bf hit@3 0.790. 미수령금 0.568→0.750(+18.2, 블랙홀 해소). 은닉 −2.3(천장 근처 RRF 재정렬 side-effect, 오분류 아님).

## 2. 공백정규화 + eval_TF 원인분석판 (0.790 → 0.793, D1 6/6)

**문제**: 대표문항 "예금자 보호 한도는 얼마인가요?"가 미적중. 또 eval.py의 D1판정이 hybrid(필터 없는 모드)로 하드코딩돼 필터 효과가 안 보였음.

**원인**: 띄어쓰기("보호 한도")가 키워드("보호한도")와 안 맞아 분류기 불명 → 필터 미적용.

**해결**: `classify_query_bf`에 공백 정규화(질의·키워드 양쪽 `replace(" ","")`). `eval_TF.py`를 ANALYZE_MODE=hybrid_bf 기준 원인분석판으로 병합 — 미적중마다 **불명/오분류/정분류** 원인 태그, 대표문항 3모드 rank 병기, `error_analysis_TF.md` 생성.

**결과**: 커버리지 70→71%, 정확도 99.1% 유지(신규 오분류 0). hybrid_bf 기준 대표 **6/6 D1 통과**, 오염 0. 잔여 미적중 62건 원인: 정분류 38·불명 22·오분류 2.

## 3. doc-count cap (0.793 → 0.857)

**문제**: 정분류 미적중의 top-3를 몬스터 문서(예금보험금이 된 selectFaqNramtAply 28청크)가 독점 → 작은 gt 문서(DpsmIbamtExpln 4청크 등)가 진입 못함.

**해결**: `_cap_by_doc()` — hybrid_bf에서 깊은 풀(max(k*5,50)) 확보 후 `parent_doc_id`당 청크 수 제한 → top-k. A/B(none/1/2/3)에서 **cap=1 최적**(hit@3 +6.4%p, hit@1 무손실=rank-1 미개입, 대표6 유지).

**선택 이유**: 결정론·저비용. 지표용 cap과 생성 맥락 분리(생성은 Parent-Child로 원문 공급 전제).

**결과**: hit@3 0.857. 은닉 1.000·채무 0.935·예보 0.907·예금보험 0.850·미수 0.795·**착오 0.722(변화 없음 — 병목으로 부상)**. 잔여 미적중 43건: 정분류 25(착오 16)·불명 16·오분류 2.

## 4. 라벨 v3 + 공용문서 필터 / 리랭킹 보류 / MMR 기각 (0.857 → 0.870)

**라벨 v3 (채택)**: 리랭킹 퇴행 진단서 잡힌 2건을 교정. 은닉 "취득경위/처리절차" 2문항에 `SprtFndCncmDclrFaq` 추가(코퍼스 실재 확인). `rag.py BF_SHARED_DOCS` — 미수령금 하드필터가 예금보험금 구비서류(`DpsmIbamtAplyPossDcmnt`)도 허용(단일 태그라 배제됐던 공용문서). → hit@3 0.857→**0.870**, hit@1 0.583, MRR 0.733, 대표6 6/6.

**리랭킹 (보류)**: 로컬 cross-encoder(`Dongjin-kr/ko-reranker`)로 hit@1 0.567→0.707·hit@3 0.913 검증 성공. **그러나 모델 2.24GB로 반복 검증이 무거워 보류** — NCP CLOVA Re-ranking 키 수령 후 `rerank_scores()` 교체로 재개 예정.

**MMR (기각)**: 타 브랜치 팀원의 유의미 결과 재현. hybrid_bf 다양성 단계를 cap↔MMR 교체(`diversify(method=)` 추상화, λ 0.3/0.5/0.7). 결과 **완패**(최고 λ에서도 hit@3 −0.143, 회복10/퇴행53). **기각 이유**: cap+하드필터가 이미 문서 다양성을 확보한 상태라, MMR은 관련 정답 청크를 밀어내고 덜 관련된 문서를 끼워넣는 순손실만 발생. 우리 문제는 다양성 부족이 아니었음. (하네스 공정성: cap@20 == ref로 검증)

## 5. 소프트부스트 전환 + 테스트셋 400 (하드→소프트)

**문제의식**: 하드필터는 분류가 틀리면(오분류) 정답을 후보에서 **아예 배제**. 커링된 400셋에서도 분류기 불명 120건(30%)·오분류 9건 → 실트래픽의 다양한 표현에선 더 위험. 키워드 커버리지 확대는 **테스트셋 오버핏** 리스크(사용자 판단으로 지양).

**해결**: 업무 신호를 "제외"가 아니라 **RRF 점수 +β 가산**으로. 전체 풀 RRF 후 같은 업무 청크에 +BF_BOOST → cap → top-k. `rag.py BF_MODE=soft|hard`(하드는 재현용 보존), `hybrid_soft()`.

**A/B**: boost@0.02가 하드 대비 파레토 우위(hit@1 +1.0pp·MRR +0.8pp, +3/−2). **채택 이유**: 동급+ 성능 + 오분류 시 정답 미배제(graceful degradation) + **리랭킹의 더 좋은 토대**(정답을 풀에 남겨 리랭커가 끌어올릴 수 있음). 한계: 오분류 자체는 못 고침(틀린 업무를 부스트) — 분류기/리랭킹 몫.

**테스트셋 400**: v3(300) + 신규 100 병합. 구조검사 통과, near-중복 0, gt 보강 2건, 업무 분포 62~72 균형.

## 6. bge-m3 전환 + RRF 재튜닝 + 팀원 라벨 병합 (→ 0.935)

**전환**: 임베딩 ko-sroberta→**BAAI/bge-m3**(1024d). `EMB_DIM 768→1024` + build 시 실제 임베딩 차원으로 인덱스 생성(모델 교체 내구성, dim assert 에러 해소). 단독 효과: 미적중 58→45, **불명 버킷 26→13**(미분류 질의도 강한 dense가 잡음 = 일반화 개선).

**과보정 발견**: 강한 dense 위에서 약한 모델용 기본값이 과보정 → dense가 1위로 뽑은 정답을 파이프라인이 끌어내림(hybrid_bf hit@1 < dense hit@1).

**재튜닝(eval_rrf_tune / eval_rrfk A/B)**:
- 가중 RRF `RRF_ALPHA=0.6`(dense 0.6/sparse 0.4)
- 부스트 완화 `BF_BOOST 0.02→0.005`(오분류 정답 재매장 방지)
- **`RRF_K 60→5`** — rrf_k 스윕서 단조 개선. **동시에 bf 장치가 여전히 유효 확인**: 같은 rrf_k=5에서 hybrid_bf 0.935 vs 순수 hybrid 0.858(**+7.7pp**) → 소프트부스트+cap+재태깅 유지 확정.

**팀원 라벨 병합**(원문 대조 검증 후): 테스트셋 gt 8건(교체 2·추가 6, 정산금/지급대행점/포상금/Fast-Track 등), selectFaqNramtAply 재태깅 팀원 버전(공통2/예보4/예금보험32, **미수 오태깅 0**). `chunk.py FAQ_BF_OVERRIDE`로 재빌드 내구성.

**"팀원 MMR 95%" 규명**: 실제 지렛대는 MMR이 아니라 **RRF(k=5)**. 팀원 업로드 error_analysis(300셋)는 재태깅 이전 구버전. 교차 브랜치 숫자 비교는 부정확 → 자기일관 내부결과로 판단. 우리 hybrid_bf+k5=0.935 > 팀원 0.930.

**결과**: hit@1 0.657 / hit@3 0.935 / MRR 0.791, 오분류 미적중 0.

## 7. 서브인텐트 패싯 부스트 (→ 0.945, 최종)

**문제**: hybrid_bf가 업무는 맞히나(오분류 0) 업무 *내부* 형제문서를 못 가림. 착오송금 hit@1 0.500(신청방법/구비서류/절차/유의사항, 송금인/수취인 유사).

**기각된 경량 시도(eval_embedinput)**: 규남 팀원 진단(동일 page_title '유의사항'이 착오송금인/수취인 두 문서에 중복→title 비변별) 검증. 문서측 임베딩 입력에 breadcrumb/sub_category 주입 → **기각**(0.925/0.932로 하락, 착오송금도 퇴행). 공통 경로가 형제를 더 붙이고 쿼리측 비대칭으로 희석. 진단은 옳으나 임베딩 트윅으론 해결 불가.

**채택(eval_facet → rag.py)**: 기존 bf 소프트부스트 위에, 질의 서브인텐트를 결정론 키워드로 잡아 매칭되는 `sub_category` 청크에만 +FACET_BOOST. (서류→구비서류, 방법/온라인→신청방법, 절차, 유의, 방문, 대상 / 방향: 보낸→착오송금인, 받은→수취인). 콘텐츠 창작 0(기존 패싯 활용), 소프트라 오발동 무해.

**FACET_BOOST 스윕**: 0.03 채택. 0.00→0.935, **0.03→0.945**(착오h3 0.833→0.889, 퇴행~0), 0.08→0.953이나 퇴행2+지배적. **선택 이유**: 0.03은 dense 1위 기여(~0.10)의 30% 수준이라 후보 내 정제선 유지. 0.08은 부스트가 검색 신호를 지배 → 오발동 시 오답 강제 상승(일반화 리스크)으로 기각.

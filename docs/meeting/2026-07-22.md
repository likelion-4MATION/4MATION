## 1. 오늘 할 일 (담당자별)
- 2회차 멘토링 보고서 작성
- 멘토링 기반 검색 방법 조사 → metadata filtering은 이미 적용되어 있음을 확인함
- 테스트셋 400개로 확장: 중복 없이, 복잡하고, 키워드가 직접 포함되지 않은 질문으로 구성 — **규남**
- API 연동 후 실제 답변 확인 — **현욱**
- MRR 재랭킹 적용 검토 — **유상**

## 2. 데이터·평가 방향 논의
**논의사항**
- 데이터 퀄리티를 어떻게 높일 것인가
- 청크(chunk) 크기를 더 작게 조정해야 하는지 검토

**논의 결과**
- 청크 크기 조정은 보류하고, 오늘은 평가(evaluation) 정비에 집중하기로 함
- 품질을 계속 다듬기보다 일단 end-to-end로 끝까지 진행해보는 방향으로 우선순위 조정
- 시나리오를 복잡하게 구성하고, 질의에 특정 키워드가 없어도 답변할 수 있도록 개선하는 방향으로 테스트셋 제작 기준을 확정함

## 3. 평가 지표 변경 — Hit@10 → Hit@3
- 현재 보유 문서가 38건에 불과해, Hit@10 기준으로는 정확도 판별이 사실상 무의미하다고 판단
- 이에 따라 평가 지표를 **Hit@3**으로 변경

## 4. 테스트셋·문서 반환 방식 논의
**테스트셋 제작 시점/방식**
- API 키로 대량 생성 시 시간이 오래 걸림 — 복잡한 질문을 생성해 돌렸을 때 답변도 복잡하게 나와야 의미가 있는데, 단순 "문서 찾기"에서 끝나면 테스트 의미가 없음
- 구성안: 복잡한 질문 100개 + 키워드 미포함(어휘가 다른) 질문 포함, 전체 규모 400개 이상

**문서 반환 방식**
- 결정 필요 사항: HTML상에서 문서를 직접 뽑아 열람 가능하게 할지 여부
- 문서를 데이터화·요약할 필요는 없음 — 중요한 건 "어떤 질문에 어떤 답이 나오는가"이지, 문서 내용 자체를 보여주는 게 아님
- 문서가 있다고 태깅되면, 고객에게 재확인 후 반환하는 방식으로 설계하는 것도 좋아보임
- API로 LLM 출력 시 답변과 함께 문서를 반환할 수 있는지 여부 추가 확인 필요

**진행 우선순위**
- End-to-end로 먼저 전체 파이프라인을 완성하고, 정확도를 끌어올리는 작업은 **실전3**에서 진행하는 것이 맞다고 판단

## 5. 리랭커(Reranker) 사용 여부
- 가벼운 모델을 사용했음에도 300개 처리에 15분 소요 (답변 1건당 약 40초)
- 정확도는 크게 상승(87% → 93%)했으나, 응답 시간 대비 정확도 개선폭이 불확실해 리랭커 추가는 최우선순위에서 제외하고 가장 나중으로 미루기로 함
- 대신 **MRR 재랭킹** 방식을 추가로 검토하기로 함 (유상 담당)

## 6. 질의 분류 방식
- 질의 분류는 정확도가 중요한 만큼, **LLM 기반 분류**로 진행하기로 결정

## 7. 실제 진행 내용 (담당자별)
- **유상**: 멘토링에서 ko-sroberta는 레거시 모델이라는 피드백을 받아 **BGE-M3**으로 임베딩 모델 교체 → 정확도 85% → 89%로 향상 확인
- **규남**: 라벨 검증 및 가벼운 임베딩 모델을 새롭게 적용해 정확도 향상 실험 진행
- **현욱**: API로 LLM 답변이 가능하도록 챗봇 데모(UI)까지 1차 구현
- **경은**: 문서가 존재하는 페이지에서 모든 문서를 개별 파일로 저장 → 이후 LLM이 답변 시 경로를 참조해 반환하도록 구조 설계

## 8. 가중치(Weight) 튜닝으로 정확도 추가 개선
- 기존에는 URL 단위로만 가중치를 부여했으나, **서브카테고리** 단위까지 확장해 부스트를 적용
- 특히 "미수령금" 항목 내에서 문서가 겹치며 발생하던 중복 문제에 가중치를 부여해 케이스를 세분화
- 위 튜닝 결과 정확도 **94.5%**까지 향상 → 이 기준으로 데모에 반영 예정

# 협업 가이드

4MATION 팀의 브랜치 전략, 커밋 컨벤션, PR 규칙을 정리한 문서.

---

## 🌿 브랜치 전략 (GitHub Flow 변형)

```text
main ──────────────●───────●────▶   배포/제출 가능한 안정 버전 (직접 push 금지)
                   ▲       ▲
develop ──●───●───●───●───●────▶   개발 통합 브랜치
          ▲   ▲       ▲
   feat/12-login      fix/17-nan-loss   작업 브랜치
```

| 브랜치 | 용도 | 규칙 |
|--------|------|------|
| `main` | 제출·배포용 안정 버전 | PR로만 병합, 직접 push 금지 |
| `dev` | 개발 통합 | 작업 브랜치의 PR 대상 |
| `feat/#-설명` | 기능 개발 | 이슈 번호 기반, 예: `feat/12-login-api` |
| `fix/#-설명` | 버그 수정 | 예: `fix/17-nan-loss` |
| `docs/#-설명` | 문서 작업 | 예: `docs/3-readme` |

**흐름**: 이슈 생성 → 브랜치 생성 → 작업 → PR (`develop` 대상) → 리뷰 1인 이상 승인 → Squash merge → 브랜치 삭제

## ✍️ 커밋 컨벤션

```text
<type>: <제목>          # 50자 이내, 한국어 OK, 마침표 X

<본문(선택)>            # 무엇을, 왜 변경했는지
```

| type | 용도 |
|------|------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 |
| `refactor` | 리팩토링 (기능 변화 없음) |
| `style` | 포맷팅, 세미콜론 등 (로직 변화 없음) |
| `test` | 테스트 코드 |
| `chore` | 빌드, 설정, 패키지 등 |

예시:

```text
feat: 로그인 API 엔드포인트 추가
fix: 학습 시 loss NaN 발생하는 문제 수정
docs: README 팀원 정보 업데이트
```

## 🔀 PR 규칙

- 제목은 커밋 컨벤션과 동일한 형식 (`feat: ...`)
- 본문에 관련 이슈 연결: `Closes #12`
- **1인 이상 리뷰 승인** 후 merge
- merge 방식은 **Squash and merge** 통일
- 충돌 해결은 PR 작성자가 담당

## 🐛 이슈 규칙

- 작업 시작 전 이슈부터 생성 (템플릿 사용)
- 라벨 활용: `feat` `bug` `docs` `question`
- 담당자(Assignee) 지정 필수

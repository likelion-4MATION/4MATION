> KDIC RAG PoC 검색 평가 신뢰도 분석 → 평가코드 보강 → 평가셋 재라벨까지의 작업 로그.
> 코퍼스·메타데이터는 손대지 않음. 변경 대상은 `eval.py`와 `data/testset_merged.jsonl` 두 개.

## 요약

평가 지표가 낮았던 원인이 검색 성능이 아니라 **평가셋 라벨 오류 + 크롤러 FAQ 페이지네이션 누락**이었음을 진단·수정해 hybrid hit@3를 **0.580 → 0.873**으로 끌어올렸다. 착오송금 0.205→0.909, 예금보험금 0.571→0.810. 지표 편향 진단용 macro·세분 지표도 평가코드에 추가.

## 1. 진단 (신뢰도 분석)

재라벨 전 hybrid hit@3 0.580이 낮았던 원인을 세 가지로 규명.

- **표본 편향**: n=150 Wilson 95% CI 폭 ±8%p. dense→hybrid 차이도 당시 표본에선 통계적으로 확정 곤란.
- **정답 쏠림**: 정답 문서 1개(`kmrsItrdAplyMthd`, 착오송금 신청방법)가 전체 정답의 25%(37건). 유효 표본이 실제로는 훨씬 작음.
- **라벨 부정합**: 착오송금 yesom 37건이 내용과 무관하게 전부 "신청방법" 한 페이지로 라벨됨 → 검색이 맞아도 오답 처리.

출처별 세분에서 yesom(키워드형, 65%)만 0.388로 지표를 끌어내리고, 그중 착오송금·미수령금이 집중적으로 실패함을 확인.

## 2. eval.py 보강

기존 micro(문항 단순평균) 지표에 다음을 추가. 기존 지표·D1 판정·오염체크는 그대로 유지.

- `group_metrics()` / `_row()`: 문항을 업무·출처별로 묶어 그룹별 hit@1/hit@3/MRR 계산.
- **micro vs macro 대비표**: macro = 6개 업무 지표의 단순평균. micro와의 gap이 곧 문항 쏠림 왜곡의 크기.
- **업무별·출처별 세분표**: 어느 업무·질의 유형이 약한지 바로 진단.

macro의 의미: macro > micro면 쏠린 대형그룹이 약해 종합점수가 눌린 것, macro < micro면 대형그룹이 쉽게 맞아 부풀려진 것.

## 3. 평가셋 재라벨 (핵심)

원칙: **질문 텍스트·스키마 불변, gt_docs만 교정.** 37개 코퍼스 페이지 전문과 대조해 실제 답이 있는 페이지로 재지정하고, 근접중복 페이지는 "허용 정답 집합"으로 확장.

150건 중 **68건 교정**.

| 업무 | 변경 건수 | 대표 교정 내용 |
|---|---|---|
| 착오송금 반환 신청 | 41 | "신청방법" 몰빵 해제 → 절차/구비서류/이의제기/이체수수료환급 페이지로 분산 |
| 은닉재산 신고 | 10 | 신고방법·절차 질문을 "신고 및 조회" 페이지로, 일부 FAQ 재지정 |
| 고객 미수령금 신청 | 7 | 용어 정의(파산배당금 등)를 상속인조회→미수령금 개요로, 인터넷신청 통합신청 추가 |
| 예금자보호제도 | 6 | FAQ 답이 개요 페이지에도 있는 경우 허용집합 확장 |
| 예금보험금 안내 | 3 | 신청/방문/인터넷을 개념(Expln)→신청절차(AplyProc)로 |
| 채무조정 안내 | 1 | "채무 확인 방법"을 채무조정제도→채무정보조회로 |

교정 예시:

- `Q72 착오송금반환지원 절차`: 신청방법 → **절차 페이지(2종)**
- `Q73 착오송금수취인 이의제기`: 신청방법 → **수취인 구비서류·유의사항**
- `Q74 착오송금수취인 이체수수료 환급신청`: 신청방법 → **수취인 구비서류**
- `Q62 예금보험금 신청`: 개념 → **신청절차**
- `Q40 파산배당금이란?`: 상속인조회 → **미수령금 개요·통합신청**

검증: 150건 유지, 질문 불변, 모든 gt가 코퍼스 내 존재, 복수정답 40건(2~4개). 정답 쏠림 37 → 26건.

## 5. 남은 과제

> 정정: 예금보험금과 미수령금은 원인이 다름. 초기 분석에서 둘을 "코퍼스 커버리지 부족"으로 묶었으나, 예금보험금 구비서류 페이지 전문 확인 결과 대부분 커버됨.

1. **예금보험금(0.571) — 검색 문제.** 구비서류 페이지(`DpsmIbamtAplyPossDcmnt`)가 본인·대리인·군복무·미성년·사망(상속)·국외거주까지 이미 커버(4청크). 라벨도 정확. 짧은 키워드 질의가 해당 청크를 top-3로 못 올리는 것이 원인 → 청킹/리랭킹(D2) 영역. 법인/단체 서류(Q114) 1건만 실제 공백.
2. **미수령금(0.389) — 커버리지/설계 문제.** 미수령금 도메인 페이지(`ProtSystNramtInqAplyNramtGudn`)에 유형별 서류 구분이 없음(상속·사망 일부만 예금보험금 Dcmnt에 혼재). person-type 변형 질문 통합 + 코퍼스 확장/재라벨 판단 필요. (평가셋 설계 영역, 보류 중)
3. **faq 누수 보정** — faq 질문이 정답 페이지에서 추출돼 1.000이 과대. 자연어화 시 페이지 문장 복붙 금지.
4. **reranker(D2)** — 위 정리 후에도 근접페이지 변별이 남으면 cross-encoder 재정렬 검토.

## 6. 크롤러 FAQ 페이지네이션 수정 (추가)

**문제**: fins FAQ 게시판이 여러 페이지인데 크롤러가 GET으로 1페이지만 수집. `.paging`의 `chgPagingNo(n)`이 `srchForm`을 POST(`curPage`=n[0-based], `pageSize`=10)로 제출하는 JS 방식이라 2페이지 이후가 누락됨.

**누락 범위**:

| FAQ | 실제 | 기존 수집 | 누락 |
|---|---|---|---|
| 미수령금통합신청 (selectFaqNramtAply) | 4p | 1p | 2·3·4p (법인/단체 답 = 3p) |
| 착오송금반환지원신청 (selectFaqMsdrGvbkAply) | 2p | 1p | 2p |
| 은닉재산신고 (selectFaqCncmPrptDclr) | 2p | 1p | 2p |

(채무 FAQ·TOP10·www FAQ는 단일 페이지라 온전)

**수정** (`crawler.py`, 백업 `crawler.py.bak`): `_collect_faq_pages()` 추가. `.paging`에서 최대 curPage를 읽어 2페이지부터 POST로 받아 `.accoWrap`의 문답(dl)을 1페이지 soup에 병합한 뒤 raw로 저장. 단일 페이지면 무변경. 오프라인 검증에서 위 3개만 다중 감지, 나머지 정상 단일 판정 확인. (KDIC가 분석환경에서 프록시 차단이라 실제 재수집은 로컬 실행 필요)

**재수집·반영 순서 (로컬)**:

```bash
cd crawler
python pipeline.py --rebuild      # 라이브 재수집(페이지네이션 반영) + 재청킹 + 재인덱싱
python eval.py                    # 지표 재확인
```

재수집 후 할 일: Q87(법인/단체 서류) 정답을 selectFaqNramtAply로 재라벨, 그 외 2·3·4p 신규 문답으로 답이 생긴 질문 있는지 점검.

**재수집 결과**: 코퍼스 청크 181→213, 미수령금 FAQ 10→38청크. 미수령금 hit@3 0.389→0.556. 단, 새 FAQ의 광범위 Q&A(저축은행 보호한도 등)가 유입되며 예금보험금은 라벨 미반영으로 0.571→0.429 일시 하락(→ 7장에서 해소).

## 7. 신규 FAQ 반영 재라벨 (2차)

**문제**: 페이지네이션 재수집으로 새 FAQ 답(법인·가지급금·person-type 서류)이 코퍼스에 생겼으나 gt가 기존 페이지만 가리켜, 검색이 더 정확한 새 답을 올려도 미적중 처리(예금보험금 0.429).

**수정** (`testset_merged.jsonl`, 백업 `.bak2`): 서류 질문 18건의 gt에 `selectFaqNramtAply`를 허용집합으로 추가 + Q87 법인/단체를 FAQ #20으로 재지정. 질문 불변, gt만 확장. 복수정답 56건.

**결과**: 예금보험금 hit@3 0.429→**0.810**, 미수령금 0.556→0.611, yesom 0.755→0.847.

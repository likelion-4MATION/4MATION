# 진행도 — 평가셋 확장 · 라벨 감사 · 청크 재태깅 · 하드필터 (2026-07-21 15:12)

> 범위: 실전1~2 경계. 평가셋 300 확장부터 business_function 하드필터까지 한 세션.
> 대상 코퍼스: dataset2 (36문서 / 215청크) · 평가셋: `testset_natural_300_v2.jsonl` (300문항)
> 한 줄 요약: 평가셋을 150→300으로 늘리고 라벨을 근거 기반으로 교정한 뒤, 오류분석으로
> "미수령금 FAQ 블랙홀 = cross-topic 오태깅" 을 규명 → 청크별 재태깅 + 하드필터로
> **hybrid hit@3 0.747 → 0.790 (+4.3%p)** 달성.

## 1. 평가셋 150 → 300 확장

- 조건: ① 업무 정규화 ② 동일 의미·다양한 표현 ③ 정답 정확 매핑 ④ 짧은 문장 ⑤ 저수준·구어체.
- 방법: 코퍼스 근거 + 기존 검증된 질문→gt_docs 매핑을 근거 은행으로 신규 150 생성(`origin_question` 명기).
- 결과: 업무 분포 평탄화(최다:최소 4.4배 → 1.8배), 신규 평균 17.7자(기존 32.0자), 완전·표기 중복 0.
- 산출: `data/testset_natural_300.jsonl`.

## 2. 라벨 감사 → v2

- 원칙: gt_docs = 본문에 답이 실제로 있는 문서 전부. 리트리버 top-3 여부가 아니라 내용으로 판단.
- 교정 23건(채무 17·착오 4·은닉 1·예금보험금 1). 대표 발견: `LbltInfoInqItrdMthdGudn`은
  제목만 "채무정보조회"이고 본문은 채무조정제도 → 조회 질의에서 제거, 제도/서류 질의에 추가.
- 결론: 오답 83건 중 라벨 탓은 ~7건, 나머지는 검색 문제 → 리트리버 개선이 본질임을 확증.
- 산출: `data/testset_natural_300_v2.jsonl` · `label_audit.md`.

## 3. 오류분석 도구

- `eval.py`에 `error_analysis.md` 자동 생성 추가: 미적중 문항 top-3에 ✅정답 / ⚠️타업무혼입 / ❌동일업무오답 태깅 + 혼입 문서 랭킹 + 업무 간 혼입 흐름.
- 보조: `compare_modes.py` (dense vs hybrid 문항별 승패), `eval_exclude.py` (문서 제외 ablation).

## 4. 진단 — 블랙홀의 정체

- `selectFaqNramtAply`(미수령금 FAQ)가 미적중 43문항 top-3에 61회 침투.
- 원인 규명: **청킹은 정상**(38 Q&A 청크). 문제는 ①이 페이지가 여러 업무 Q&A를 한데 담은
  cross-topic 게시판인데 URL 규칙상 업무 1개(미수령금)로 태깅 → 청크 100% 오태깅,
  ②38청크 = 검색 인덱스에서 "복권 38장" → 문서 크기 불균형.
- FAQ 전수 점검: 오염은 `selectFaqNramtAply`(100%)·`selectFaqTop10`(100%, 공통→실제 착오)에 집중.
  주제별 FAQ(예보·은닉·착오)는 0% 정상.

## 5. 청크별 business_function 재태깅 (chunk.py)

- 결정론적 키워드 규칙(LLM 미개입, 관리자 트리거 재생성 요건 충족).
- 이중 안전장치: (1) 다른 업무가 원 태그보다 margin 2 이상 우세할 때만 재태깅,
  (2) 문서 재태깅 비율 50% 미만이면 통째로 원 태그 유지(깨끗한 문서 보호).
- 결과: 문서 38개 중 2개만 변경, 본문·chunk_id 무손실.
  - `selectFaqNramtAply`: 미수령금 38 → 예금보험금 28 · 예보 6 · 미수령금 4
  - `selectFaqTop10`: 공통 10 → 착오송금 9 · 공통 1
- 인덱스 반영: 텍스트 불변이라 재임베딩 없이 `chunk_meta.jsonl`의 태그만 갱신(정렬 유지).

## 6. business_function 하드필터 (rag.py)

- `classify_query_bf()` — 질의 전용 키워드 분류(결정론적). 커버리지 70% · 정확도 99%.
  불명이면 None → 필터 미적용(전체 검색)으로 안전 폴백.
- `hybrid_bf` 모드 신규(옵트인, 기존 hybrid 보존): 예측 업무 청크로만 검색 범위 제한.

## 7. 결과

| mode | hit@1 | hit@3 | MRR |
|---|---|---|---|
| dense | 0.437 | 0.700 | 0.576 |
| hybrid | 0.557 | 0.747 | 0.670 |
| **hybrid_bf** | **0.563** | **0.790** | **0.688** |

micro≈macro(0.790/0.792)로 쏠림 없는 전반 개선. 미적중 76 → 63 (약 13문항 회복).

| 업무 | hybrid → hybrid_bf |
|---|---|
| 고객 미수령금 | 0.568 → **0.750 (+18.2%p)** ★ 블랙홀 해소 |
| 채무조정 | 0.848 → 0.891 (+4.3) |
| 예금보험금 | 0.625 → 0.675 (+5.0) |
| 예금자보호 | 0.833 → 0.852 (+1.9) |
| 착오송금 | 0.708 → 0.722 (+1.4) |
| 은닉재산 | 0.886 → 0.864 (−2.3) |

- 은닉 −1문항: 오분류 아님(검증). 필터 시 후보 풀 20→전체로 넓어지며 생긴 RRF 재정렬 side-effect(천장 근처라 노출).

## 8. 남은 과제

1. **공백 정규화** — 대표문항 "예금자 보호 한도는…"이 띄어쓰기 때문에 분류기 불명 처리 → 필터 미적용. 질의 공백 제거 매칭으로 즉시 해결 가능(예보 커버리지도↑).
2. "개산지급금 정산금" 2건 오분류(미수령금→예금보험금) — 경계 어휘 가중 조정.
3. 은닉 −1문항 — hybrid_bf pool 크기 일치 재실험(선택).
4. (선택) 소프트 부스트 vs 하드필터 비교.

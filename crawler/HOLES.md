## H4. robots.txt를 urllib.robotparser가 못 읽음 (전 페이지 unreachable 기록)

- **막힘**: 크롤러 로그에 `robots.txt 읽기 실패 → 보수적으로 진행`. 메타의 robots_status가 전부 `unreachable`로 기록됨.
- **실측**: `requests`로 직접 받으면 두 호스트 모두 **HTTP 200**. 즉 robots.txt는 실재. urllib이 실패하는 원인은 기본 UA(Python-urllib) 차단/넷퍼넬 트래픽제어 응답으로 추정.

  - fins: `Disallow: /cm/bbs/`, `/cm/srch/selectItgrSrch.do` (User-agent: Googlebot)
- **우회**: 실측 robots 규칙을 config.POLICY_DISALLOW 오버레이로 하드코딩해 경로 패턴으로 명시 차단(H5). www 필수 URL은 List.do/Dtl.do/srch 미해당 → 정상 수집.
- **[D1 0-a 해결]** robots_status를 `requests.get(UA=config.USER_AGENT)`+`parse()`로 근본 수정 → 두 호스트 robots.txt 정상 파싱(200), 정상 페이지 `unreachable`→`allowed`. GET 실패 시 fail-open 금지(unreachable 유지+오버레이 의존). 규칙이 Googlebot 전용이라 우리 UA엔 can_fetch=allowed이므로 **POLICY_DISALLOW 오버레이는 이중 안전장치로 존치**(fins /cm/bbs/ 차단 담당). DECISIONS D1 참조.

- **막힘**: 이번 세션에서 `build_yesom_testset.py`를 처음 실행하니 예솜24 응답노드가 매번 0건. 두 가지 독립된 원인이 겹쳐 있었다.
  1. macOS python.org 배포판이 `cert.pem`을 설치하지 않아 `urllib`이 `CERTIFICATE_VERIFY_FAILED`로 매 요청 실패(같은 머신의 `curl`·`requests`는 자체/certifi 번들을 써서 영향 없음 — `crawler.py`가 `requests`를 쓰는 이유로 이 문제를 겪지 않음).
  2. 인증서를 우회해도 `urllib` 기본 User-Agent(`Python-urllib/3.x`)로 보낸 POST는 KDIC WAF가 `DMZ WAF` 오류 HTML(EUC-KR)로 되돌려 JSON 파싱 전 단계에서 조용히 스킵됨(`ask()`의 예외 처리가 이를 삼켜 노드가 fallback-only로 보임).
- **해결**: 실행 시 `SSL_CERT_FILE=$(python3 -c "import certifi; print(certifi.where())")` 지정 + `build_yesom_testset.py`의 `ask()`에 `config.USER_AGENT`(기존 크롤러와 동일 식별 UA)와 `Referer: https://pubkbot.kdic.or.kr/chatbot.html?source=homepage` 헤더 추가. 이후 98노드 정상 수집.
- **팀 이관**: 신규 로컬 환경에서 이 스크립트를 처음 돌릴 때 위 두 증상이 재현되면 같은 조치 적용. `requirements.txt`에 `certifi` 명시 여부는 팀 판단(현재는 다른 의존성의 이행 종속으로 이미 설치되어 있어 즉시 동작은 함).

## H20. (신규) 재실행 시 질문 라벨 문구가 달라짐(set 순회 비결정성) — 열림(경미)

- **발견**: 첨부받은 `testset_yesom.jsonl`(ref)과 이번 세션 재실행 결과를 비교하니 98건 세트는 동일하지만 약 25건의 **질문 문구**가 다르다(예: "은행" ↔ "보호 금융상품 - 은행", "예금자 본인 신청서류 (미수령금 신청서류)" ↔ "예금자 본인 신청서류 (보험금신청서류)"). `(gt_docs, business_function)` 멀티셋은 완전히 동일 — 의미상 오류는 아니고 **라벨 disambiguation 순서**만 달라짐.
- **원인 추정**: `crawl()`의 BFS 큐 `queued`가 **`set`**이라 `list(queued)` 순회 순서가 파이썬 프로세스마다(PYTHONHASHSEED 기본 랜덤) 달라짐. 같은 라벨("은행" 등 GENERIC 항목, 서류 세부 노드)이 여러 상위 경로에서 도달되는 경우 `build()`의 `while q in seen` 중복 처리 순서가 실행마다 바뀌어 어떤 경로가 "먼저"(접미사 없이) 잡히는지가 갈림.
- **영향**: gt_docs·business_function 매핑(즉 D1 통과/미통과 판정, 대표 6문항 5/6)은 안정적. 다만 **eval 수치는 완전히 무관하지 않음** — 질문 라벨 문구 자체가 검색 쿼리로 쓰이므로, 문구가 바뀌면 dense/hybrid 검색 결과도 미세하게 달라진다. 실측: `build_yesom_testset.py`→`pipeline.py`→`eval.py`를 연속 2회 재실행했더니 hybrid hit@1 0.400→0.393, hit@3 0.587→0.580, MRR 0.511→0.505로 **약 0.5~0.7pt 변동**(D1 판정 결론은 양쪽 다 5/6 통과로 동일). 리포트 수치를 소수점 단위로 비교·인용할 때는 이 변동폭을 감안할 것.
- **팀 이관**: 완전한 재현성이 필요하면 `crawl()`의 `queued`를 `set` 대신 순서 보존 자료구조(list + 존재 체크, 또는 `dict.fromkeys`)로 교체 권장. D1 판정 자체엔 영향 없어 급하지 않으면 보류 가능.

## H21. (신규) crawl() 패스마다 전체 큐 재조회 — 실행시간 O(pass×n) — 해소

- **막힘**: 팀원이 로컬에서 `build_yesom_testset.py`를 실행했더니 5분 넘게 끝나지 않아 Ctrl+C로 중단. 단발 요청 진단(`ask()` 단독 호출)은 0.13초·200 정상 응답이라 네트워크·SSL·WAF 헤더 문제는 아니었음.
- **원인**: 원래 `crawl()`의 바깥 `while changed` 루프가 **패스마다 `queued` 집합 전체**를 다시 조회했다. 이미 버튼을 확보해 더 알아낼 게 없는 노드까지 매 패스 반복 조회되어, 노드가 발견될수록 패스당 요청 수가 계속 커짐(예: 6→28→53→82→98→98(최종 확인) 순으로 매번 전체 재조회 → 누적 방문 300회 이상). 개별 요청은 빨라도(≈0.1~0.3초) 총량이 누적되며 특히 네트워크 지연이나 WAF의 완만한 스로틀링이 겹치면 5분 이상으로 불어날 수 있음.
- **해결**: `to_visit`을 "이번 패스에 새로 발견된 노드" + "버튼을 못 얻고 fallback만 본 노드(최대 3회 추가 재시도)"로만 한정하도록 재작성, 이미 버튼을 확보한 노드는 이후 패스에서 제외. 진행률 print도 추가(`[예솜 크롤 pass N] 조회 …개 · 누적 발견 …개`)로 실행 중임을 눈으로 확인 가능.
- **[상태 07-15]**: **해소** — 재실행 결과 5패스·66초에 동일한 98개 노드(업무별 분포 동일) 완료. 기존 방식이면 같은 결과를 얻는 데 노드 재조회가 누적 300회 이상 필요했을 것을 98회로 축소(≈3배 이상 절감).
